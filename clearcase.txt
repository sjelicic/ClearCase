The popular task in ClearCase is first to make a type of something and then to use the something.


Make Type 			Attach 				Comment
Command 			Command
		
mklbtype			mklabel				Labels are used to mark a particular version of the element (e.g., file or directory). The most 
										common use for them is to mark a set of particular versions of files and directories that represent a snapshot of a project at some milestone (e.g., a release). Since you are not allowed to assign the same label to two different versions of the same element, labels ensure that creation of new versions of elements in the future does not affect the set of versions marked with a label.

mkattype			mkattr				Attributes are the name/value pairs. The value can be of different types (e.g., an integer, a string 
										of characters or a boolean value TRUE/FALSE). When you create an attribute type with a mkattype command you need to specify what the values can be. The attributes are very flexible and you can attach them to essentially any object on the VOB, i.e., to an element, to a particular element version, to a branch, etc. You can use them to add some notes or assign some property to the VOB object. You can use the ClearCase ct find ... command and find the elements that have certain attribute values. To tell you the truth, I never used attributes directly.

mkhltype			mkhlink				Hyperlinks connect a VOB object to another object. Obviously the two objects have to be of the same 
										kind before they can be hyperlinked. The merge arrow is an example of the hyperlink of type Merge that comes predefined with ClearCase. The hyperlinks are logical associations. For example, you can link an element version to a file that describes what has been done.

mkeltype			mkelem or			The element type represents a type of file that is created. ClearCase comes with some eltypes 
					chtype				predefined, like text_file or directory. The element type is usually assigned to the element 							automatically based on the magic file that associates extensions of files with the type. 

mkbrtype			mkbranch			The branch is a VOB object that collects consecutive versions of the element. The main branch is 
										predefined in ClearCase. However, if you want to create a new sub-branch, you need to first create it with the mkbrtype command. This is done usually when you create a view which will be placing new element versions on a specific branch according to the rules in the config spec file. Then the config spec file instructs the ClearCase to use mkbranch command to assign the branch to a version of an element. Now you see what I am talking about... The lingo... ##########################################################################################################

Creating streams, activities and elements

Create new element
	> ct mkelem –eltype <element-type> <element-name>.<extension> 

Create new stream
	> ct mkstream –in <parent-stream> [-baseline <baseline-name>] –nc <stream-name>

Create view on stream
	> ct mkview –tag <view-tag> -stream <stream-name@pvob> -stgloc <view-storage-location>

Create new activity
	> ct mkact  <activity-name>


Listing activities and streams

Look at information related to your current stream
> ct lsstream –l 
> ct desc –l stream:<stream-name>@<pvob>

Find the activities associated with your view
> ct lsact 

Find the name of the current activity
> ct lsact –cact

Find the changeset related to your activity
> ct lsact –l


Listing any UCM object

List the UCM component in PVOB
	> ct lscomp -invob <pvob>

List the existing baselines for a given component 
	> ct lsbl -comp <component-name>@<pvob> [-stream <stream-name>@<pvob>]

Cleartool describe provides verbose information for all UCM objects:
	> ct desc -l …
	component:<component-name>@<pvob>
	stream:<stream-name>@<pvob>
	baseline:<baseline-name>@<pvob>
	activity:<activity-name>@<pvob>
	project:<project-name>@<pvob>
	vob:<vob-tag>



Listing UCM objects and checkouts
You can also describe elements stored inside a VOB as well as change-sets. 
You can use instead also the ls* commands like
> ct …
	lscomponent -l <component-name>@<pvob>
	lsstream -l <stream-name>@<pvob>
	lsbaseline -l <baseline-name>@<pvob>
	lsactivity -l <activity-name>@<pvob>
	lsproject -l <project-name>@<pvob>

Determine private files and ClearCase elements 
	> ct ls
Which elements are checked out currently in my view?
> ct lsco –me
Which elements are checked out currently in the path?
> ct lsco –all <vob-tag>


Views

Find the Global Storage locations
	> ct lsstgloc -view

Create a view, set view and delete a view
> ct mkview -tag <view-tag> -stream <stream-name@pvob> -stgloc <view-storage-location>
> ct setview <view-tag> (startview on Windows)
> ct rmview -tag <view-tag>

Error during view removal?
> ct rmview -avobs –uuid <uuid-of-the-view>
> ct unregister -view –uuid <uuid-of-the-view>
> ct rmtag -view <view-tag>

Don’t forget to remove the .vws directories of the view you removed.

Find view-private files: ct lsprivate


Checkout and check-in

Checkout current directory: ct co –nc .
Check out an element within your view: ct co –nc <relative-path> or <full-path>
Check in an element within your view: ct ci –nc <relative-path> or <full-path>
Check-in current directory: ct ci –nc .
Checked out the wrong file, or don’t want to save changes? ct unco <relative-path> or <full-path>


Rebase

Rebase dev stream to a specific baseline, for modifiable and non-modifiable component only
	> ct rebase -baseline <baseline-name>@<pvob>
Rebase to recommended baseline from project level
> ct rebase -rec
Preview rebase
	> ct rebase -preview
Finish rebase
	> ct rebase -complete


Deliver

Deliver from development stream to integration stream
> ct deliver [-to <integration-stream-view-tag>]
> ct deliver -complete

Cancel delivery
> ct deliver –cancel

Preview delivery
	> ct deliver –preview


Compare content of streams and baselines

You can compare Stream with Stream, Baseline with Baseline and Stream with Baseline

Baseline vs Baseline:
	> ct diffbl baseline:<baseline1>@</pvob> baseline:<baseline2>@</pvob> 
Stream vs Stream:
	> ct diffbl stream:<stream1>@</pvob> stream:<stream2>@</pvob> 
Stream vs Baseline:
	> ct diffbl stream:<stream>@</pvob> baseline:<baseline>@</pvob> 
Differences between 2 versions of a file
	> ct diff <element>@@<branches>/<versionX> <element>@@<branches>/<versionY>
